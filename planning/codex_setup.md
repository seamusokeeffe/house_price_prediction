# Codex Setup Guide

## Recommendation
Use a simple prompt-driven Codex setup for now:

- one root `AGENTS.md`
- specialist prompt files in `/prompts`
- durable planning and reference docs under `/planning` and `/docs`
- no custom skills or subagents until the project has repeated, stable workflows

This is the best fit for a solo, local-first ML project because the current risk is unclear scope and weak planning, not lack of automation.

## Root AGENTS.md vs layered AGENTS.md

Keep only the root `AGENTS.md` at this stage.

The root file should contain:

- locked project decisions
- working mode and repo priorities
- directory conventions
- specialist workflow rules
- output expectations
- constraints

Add subdirectory `AGENTS.md` files later only if one area needs special operating rules. Likely candidates:

- `/src/AGENTS.md` when implementation begins and coding style stabilises
- `/data/AGENTS.md` if local data handling needs strict privacy or naming rules
- `/artifacts/AGENTS.md` if model/report artifact versioning becomes easy to mishandle

Do not add layered instruction files just to restate the root decisions.

## What belongs where

Durable instructions belong in `AGENTS.md` when they should affect almost every future task.

Specialist prompts belong in `/prompts` when they describe a role, scope, and deliverables for one workstream.

Planning docs belong in `/planning` when they are working plans, roadmaps, decisions, assumptions, or handoff notes.

Durable specs belong in `/docs` when they should outlive a planning phase and guide implementation.

Experiment outputs belong in `/artifacts` when they are generated by runs or reports. Planning summaries of those runs belong in `/planning/modelling/experiment_log.md`.

## Specialist prompt integration

Use the five specialist prompts as role briefs:

- System Architect: update `/planning/system_architecture.md`, `/planning/roadmap.md`, and `/planning/decision_log.md`
- Data Specialist: create or update `/planning/data/*` and `/docs/data/*`
- Modelling Specialist: create or update `/planning/modelling/*` and `/docs/modelling/*`
- Product Specialist: create or update `/planning/product/*` and `/docs/product/*`
- Deployment Specialist: create or update `/planning/deployment/*` and `/docs/deployment/*`

Invocation convention:

```text
Execute @prompts/data_specialist.md
```

Before writing, the specialist should inspect:

- `AGENTS.md`
- `/planning/decision_log.md`
- `/planning/roadmap.md`
- existing files in its target directory
- `/planning/specialist_handoffs.md`

To avoid redoing work:

- treat locked decisions as inherited constraints
- update existing files before creating new ones
- add unresolved dependencies to `/planning/specialist_handoffs.md`
- add new cross-cutting decisions to `/planning/decision_log.md`

## Skills and subagents

Use plain prompts only for now.

Add a Codex skill later when a repeated workflow has a stable recipe, for example:

- running an experiment and producing a standard summary
- validating a new raw data extract
- generating a valuation report from saved model artifacts

Add subagents later only if parallel review becomes valuable, for example:

- one agent evaluates data leakage while another reviews feature definitions
- one agent reviews report clarity while another checks deployment steps

For the next few weeks, custom skills and subagents would add more process than leverage.

## Working conventions

Planning documents:

- use clear names, not dates, for durable workstream plans
- use dated entries only inside logs
- keep "must / should / nice-to-have" priority language
- record assumptions explicitly when facts are unknown

Decision log:

- add decisions as `D-###`
- keep old locked decisions when superseded
- add a new decision that states what changed and why
- do not silently edit the meaning of locked decisions

Assumptions log:

- add assumptions when work depends on unknown information
- include owner, status, and validation path
- promote assumptions to decisions only after review

Experiment log:

- create one entry per meaningful experiment
- link config, data snapshot, artifact path, and summary metrics
- always include segmented failure notes

Handoffs:

- use `/planning/specialist_handoffs.md` for cross-workstream dependencies
- each entry should name the source specialist, target specialist, question, and status

## First setup tasks

1. Keep the root `AGENTS.md` as the only durable instruction layer.
2. Use `/prompts` as the specialist brief library.
3. Create the assumptions, handoff, and experiment logs before deep planning.
4. Run the five specialist prompts once, in this order: system, data, modelling, product, deployment.
5. Revisit `AGENTS.md` only after the first implementation sprint reveals repeated friction.

## Defer

- custom Codex skills
- formal subagents
- layered `AGENTS.md` files for every directory
